在Go语言并发处理中，singleflight和Redsync是解决不同场景下资源竞争和重复计算问题的两种方案，它们的适用场景和实现原理有较大差异：

1. 单机下的singleflight
#

singleflight是Go标准库sync包中的一个扩展工具（通常通过golang.org/x/sync/singleflight实现），用于在单机单进程环境下解决并发请求的重复计算问题。

核心功能：

  • 当多个goroutine同时请求同一个资源时，只让第一个请求实际执行，其他请求等待结果返回后直接复用该结果
  • 有效防止缓存击穿、避免重复计算或重复数据库查询

实现原理：

  • 内部维护一个映射表，记录当前正在处理的请求键值
  • 使用等待组(WaitGroup)和互斥锁保证并发安全
  • 第一个请求执行完成后，通过channel通知所有等待的goroutine

适用场景：

  • 单机环境下的缓存穿透防护
  • 避免重复计算或重复IO操作
  • 减轻数据库等后端服务的瞬时压力

示例代码片段：

import (
    "golang.org/x/sync/singleflight"
)

var g singleflight.Group

func fetchData(key string) (interface{}, error) {
    // 多个goroutine同时调用时，只会实际执行一次
    v, err, shared := g.Do(key, func() (interface{}, error) {
        // 实际的获取数据逻辑，如数据库查询
        return queryFromDB(key)
    })
    if shared {
        // 表示当前结果是复用的，不是本次调用实际执行的
        fmt.Println("result is shared")
    }
    return v, err
}

2. 分布式锁Redsync
#

Redsync是基于Redis实现的分布式锁库（go-redsync/redsync），用于在分布式系统中实现跨进程、跨机器的资源同步。

核心功能：

  • 在分布式环境中提供互斥锁机制，保证同一时刻只有一个进程能访问共享资源
  • 基于Redis的SETNX命令和过期机制实现，支持自动续期
  • 支持多个Redis实例，提高锁服务的可用性

实现原理：

  • 利用Redis的SET key value NX PX命令实现锁的获取
  • 采用红锁算法(Redlock)，通过多个独立Redis实例提高可靠性
  • 锁的释放需要验证持有者标识，防止误释放
  • 支持自动续期机制，避免长时间任务执行时锁过期

适用场景：

  • 分布式系统中的资源竞争控制
  • 跨服务的任务调度，避免重复执行
  • 分布式环境下的临界区保护
  • 分布式系统中的 Leader 选举

示例代码片段：

import (
    "github.com/go-redsync/redsync/v4"
    "github.com/go-redsync/redsync/v4/redis/goredis/v9"
    "github.com/redis/go-redis/v9"
)

// 创建Redis客户端
client := redis.NewClient(&redis.Options{
    Addr: "localhost:6379",
})
pool := goredis.NewPool(client)

// 创建redsync实例
rs := redsync.New(pool)

// 获取锁
mutex := rs.NewMutex("resource-lock")
if err := mutex.Lock(); err != nil {
    // 处理获取锁失败的情况
    panic(err)
}

// 执行业务逻辑
defer func() {
    // 释放锁
    if _, err := mutex.Unlock(); err != nil {
        panic(err)
    }
}()

两者对比与选择建议
#

特性 singleflight Redsync
适用范围 单机单进程 分布式系统
解决问题 重复计算/请求 跨进程资源竞争
依赖 无外部依赖 需要Redis服务
性能 极高（内存操作） 受网络和Redis性能影响
可靠性 进程内可靠 依赖Redis集群可靠性

选择建议：

  • 单机环境下的并发优化，优先使用singleflight
  • 分布式系统中需要跨服务同步时，使用Redsync
  • 可以组合使用：在每个节点内部用singleflight减少本地重复请求，同时用Redsync保证分布式环境下的全局唯一性

实际应用中，需要根据系统架构、并发量和数据一致性要求来选择合适的方案。
